Secrets Management

Secrets Management refers to the process of securely storing, managing and accessing sensitive information such as API keys, passwords, tokens, certificates and other confidential data.

  1. üõ°Ô∏è Protection The goal is to protect these secrets from unauthorized access while ensuring they are available to authorized applications and services when needed.

  2. üîí Security Effective secrets management is crucial for maintaining the security and integrity of applications.

Traditional Identity Management‚Äã

Traditionally, secrets like tokens, API keys and passwords were embedded directly into application code, stored in configuration files or environment variables.

  1. ‚ö†Ô∏è Security Risks This approach posed significant security risks, as secrets could easily be exposed through source code leaks, configuration mishandling or unauthorized access to the storage locations.
Remediation with Azure Key Vault üîë

To remediate this situation, we use Azure Key Vault to securely store and manage secrets, encryption keys and certificates.

  1. üõ°Ô∏è Centralized Management Azure Key Vault provides a centralized and secure way to manage access to secrets, reducing the risks associated with hardcoding or manually managing sensitive information.

  2. üîÑ Access Management While developers can securely store the secrets in Azure Key Vault, services still need a way to access Azure Key Vault by using API Keys or tokens.

Recommended way of Identity Management‚Äã

  1. The recommended way for managing identity now is Azure Managed Identity

  2. Azure Managed Identity is a service that simplifies secrets management by automatically managing the identity for Azure resources, eliminating the need to store credentials in code.

  3. Managed Identity provides a secure and streamlined way for Azure services to authenticate to other services that support Entra ID (Azure AD) authentication.

Types of Azure Managed Identity‚Äã

TypeDescription
System-Assigned Managed Identity

Automatic Creation: Automatically created and managed by Azure for a specific single resource (e.g., a virtual machine or Azure function).

Lifecycle Tied: The identity is tied to the lifecycle of the resource; it is created when the resource is created and deleted when the resource is deleted.

Ideal Use: Ideal for scenarios where the identity is only needed for a specific resource and you want Azure to manage it automatically.

User-Assigned Managed Identity

Independent Creation: Created as an independent Azure resource that can be assigned to multiple Azure resources.

Independent Lifecycle: The identity exists independently of the resources it‚Äôs assigned to, allowing for greater flexibility and reuse.

Ideal Use: Useful when you need a consistent identity across multiple resources or want to manage the identity independently of the associated resources.

Recommendation‚Äã

  1. It is highly recommended that only Managed Identity be used for Azure resources.

  2. Use traditional methods such as tokens or API keys only when there is a specific, valid reason that Managed Identity cannot fulfill.

  3. Examples
    1. Legacy systems that do not support Entra ID authentication.
    2. Third-party services that require API keys.
